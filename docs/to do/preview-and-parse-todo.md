# 预览与解析能力 To Do

这份文档讨论的不是“再多支持几个平台”这么简单，而是把幕库从“任务提交器 + 本地文件产物”往前再推一步，变成一个更完整的内容工作台：

- 提交前能先解析和预览源内容
- 提交后能直接在网页里预览逐字稿、解析稿、知识库稿
- 用户不需要每次都回到文件系统里找产物

这条路线很适合幕库，因为你本来就不只是下载器，而是 `Video-to-Markdown / Knowledge pipeline`。

## 当前推进状态

截至 `2026-04-16`：

- `已完成` Phase 1 的核心骨架：Web 任务详情里可直接预览 `原始逐字稿 / 逐字稿 / 解析稿 / 知识库稿 / 转写信息`
- `下一步` 进入源内容解析预览，也就是这里定义的 `POST /api/parse` 或等价能力

所以这份文档后面的重点，已经从“先做产物预览”切换成“继续做任务前解析预览 + 内容页能力深化”。

## 先说结论

这个需求值得做，而且很适合做成幕库的下一阶段核心体验。

它的价值不在于“更像万能下载器”，而在于：

- 让用户在处理前先判断值不值得入库
- 让用户在处理后直接消费内容，而不是只拿到文件路径
- 为后续 API 化、更多平台接入、内容页分享和轻量协作打基础

整体难度属于：

- `预览已有产物`：低到中
- `源内容解析预览`：中
- `抽象成可扩展 provider + 通用预览 API`：中到高

如果只做一个 MVP，难度并不夸张；真正难的是把它做成未来可扩的平台底座。

## 需求拆解

建议把“预览”拆成两层，不要混在一起。

### 1. 源内容预览

这是任务开始前的预览。

目标是用户粘贴链接后，不要立刻开跑，而是先看到：

- 平台
- 标题
- 作者 / 频道
- 缩略图 / 封面
- 时长
- 可选格式
- 是否存在字幕
- 预计走哪条链路
  - 字幕直提
  - 音频转写
  - 平台专用 provider
- 风险提示
  - 需要 cookies
  - 平台限制较强
  - 该链接可能是合集 / 列表而不是单条内容

这部分最像“解析预览”。

### 2. 产物内容预览

这是任务完成后的预览。

目标是用户在 Web UI 里直接看：

- 原始逐字稿
- 清洗后的逐字稿
- 解析稿
- 知识库稿
- 转写 metadata

并且可以直接：

- 切换不同稿件
- 复制 Markdown
- 下载单个文件
- 打开产物目录

这部分最像“单篇文章预览下载”，也是你刚刚提到的重点。

## 为什么值得做

### 产品价值

- 现在 Web 端更像任务面板，不像内容工作台
- 用户知道“任务完成了”，但还得自己去找文件
- 如果能直接预览单篇稿件，产品完成度会明显上一个台阶

### 对定位有帮助

这会强化幕库“知识沉淀工具”的定位，而不是削弱它。

因为真正的知识产品不应该只返回：

- 下载路径
- 文件名
- 完成状态

它应该直接返回“可阅读、可判断、可继续处理”的内容页面。

### 对后续 API 化有帮助

一旦预览能力抽出来，你后面自然就可以长出：

- `POST /api/parse`
- `GET /api/jobs/:id`
- `GET /api/artifacts/:id`
- `GET /api/preview/:id`

这会让 CLI、Web、Skill、以后可能的浏览器插件或移动端都更容易共用同一套后端。

## 当前现状

基于当前仓库，已经有不少基础可以复用：

- 已有平台识别：`detect_platform()`
- 已有字幕探测和字幕优先策略
- 已有 sidecar 产物规范
- 已有 `artifacts` / `knowledge` / `capture` CLI 能力
- Web 已有任务列表和详情面板

当前短板主要在于：

- Web 缺少单独的“解析预览”接口
- Web 详情主要显示路径和状态，不显示正文内容
- 平台能力主要还在 `app.py` 里，后续扩展平台会越来越重
- 下载、转写、知识整理耦合得比较紧，轻量预览不容易单独调用

## 建议的目标能力

建议把目标定成 4 个接口层能力。

### A. 输入解析层

新增一个轻量解析入口，例如：

`POST /api/parse`

输入：

- 原始 URL
- 分享文案

输出：

- `normalized_url`
- `platform`
- `content_type`
- `title`
- `author`
- `thumbnail`
- `duration`
- `formats`
- `subtitle_candidates`
- `auth_required_hint`
- `transcript_route_prediction`
- `warnings`

这一层只负责回答：

“这是什么内容？值不值得处理？大概率怎么处理？”

### B. 任务结果层

扩展任务详情接口，让它除了返回路径，也返回更完整的产物索引。

例如：

- `raw_path`
- `markdown_path`
- `article_path`
- `knowledge_path`
- `meta_path`
- `preview_ready`

### C. 内容预览层

新增内容预览接口，例如：

`GET /api/preview?path=...`

或者更稳妥一点：

`GET /api/artifacts/:artifact_id/preview?kind=article`

支持：

- `raw`
- `transcript`
- `article`
- `knowledge`
- `meta`

返回可以是：

- 原始文本
- Markdown 文本
- 渲染后的 HTML

### D. 下载导出层

除了本地文件产物，还要支持单文件下载入口，例如：

- 下载逐字稿
- 下载解析稿
- 下载知识库稿
- 下载 metadata

这样用户不一定非得进入文件夹。

## 推荐实现方案

## Phase 1：先做产物预览

这是最值得先做的一步。

状态：`已完成`

原因：

- 代码复用最多
- 风险最低
- 用户感知最强

### To Do

- 新增统一的 artifact 读取函数
- 从现有 sidecar 推导可预览的稿件列表
- 新增预览接口，按类型返回文本内容
- 在 Web UI 任务详情里增加 tab
  - 逐字稿
  - 解析稿
  - 知识库稿
  - metadata
- 在 UI 中做 Markdown 渲染
- 增加“复制 Markdown”和“下载当前稿件”按钮
- 对不存在的稿件显示明确状态

### 难度评估

- 技术难度：`低到中`
- 架构风险：`低`
- 用户价值：`高`

这一步很像把你已经生成好的内容“重新接回网页”。

## Phase 2：再做源内容解析预览

这一步是从“任务提交器”升级到“解析工作台”的关键。

### To Do

- 抽出独立的 `parse_source()` 能力
- 把当前 preview / subtitle probe / metadata 提取逻辑从任务执行链里拆出来
- 新增 `POST /api/parse`
- 支持返回：
  - 标题
  - 平台
  - 作者
  - 封面
  - 时长
  - 格式列表
  - 字幕可用性
  - 推荐处理链路
- 区分单条内容和合集 / 列表 / 频道
- 前端新增“先解析再开始”流程
- 解析成功后允许用户：
  - 选择只下载
  - 选择提取逐字稿
  - 选择继续生成知识库

### 难度评估

- 技术难度：`中`
- 架构风险：`中`
- 用户价值：`高`

难点不在前端，而在于把当前散落在下载流程中的信息采集逻辑拆成稳定接口。

## Phase 3：补齐 provider 抽象

如果后面你真要扩展更多平台，这一步迟早要做。

### To Do

- 定义 provider 协议，例如：
  - `normalize_input()`
  - `parse()`
  - `download()`
  - `extract_subtitles()`
  - `get_direct_url()`
- 保留 `yt-dlp` 作为默认 provider
- 把 `Douyin` 做成专用 provider
- 后面新增平台时优先做 provider，不继续在 `app.py` 里写 if/else
- 把 provider 输出统一成同一份 schema

### 难度评估

- 技术难度：`中到高`
- 架构风险：`中`
- 长期价值：`极高`

这一步不是为了马上让 UI 更好看，而是为了避免未来平台一多就失控。

## Phase 4：把预览做成真正的内容页

这是“下载器”变“内容工作台”的最后一步。

### To Do

- 为单篇稿件提供独立预览页
- 支持 URL 直达：
  - 某条任务
  - 某份解析稿
  - 某份知识库稿
- 增加目录导航
- 增加源链接、作者、平台、转写路线、模型等信息头部
- 支持全文搜索
- 支持一键复制 / 导出 Markdown
- 后续可考虑：
  - 分享只读链接
  - 多稿对照
  - 历史版本

### 难度评估

- 技术难度：`中`
- 产品收益：`高`
- 适合放在前 3 个阶段稳定后再做

## 关键技术点

### 1. 预览不应该直接暴露任意本地路径

这点很重要。

不要让前端直接传任意文件路径给后端读取。更稳妥的做法是：

- 基于任务 id
- 基于 artifact id
- 或者只允许读取受控目录下的 sidecar 文件

否则后面很容易踩路径遍历和权限边界问题。

### 2. Markdown 和纯文本要分开处理

建议区分：

- 原始逐字稿：纯文本
- 逐字稿 / 解析稿 / 知识库稿：Markdown
- metadata：JSON

前端渲染逻辑也按类型分别处理，不要全部当字符串硬贴。

### 3. 解析接口和执行接口要分开

不要把 `parse` 和 `start` 混成一个接口。

更好的结构是：

- `POST /api/parse`
- `POST /api/start`
- `GET /api/tasks`
- `GET /api/artifacts/:id`
- `GET /api/preview/:id`

这样体验和扩展性都会更稳。

### 4. 预览能力会自然推动缓存设计

一旦有了解析预览，最好缓存这些信息：

- 标题
- 封面
- 作者
- 字幕可用性
- 格式列表

这样用户切回页面时不需要每次重新探测。

## 风险与瓶颈

### 风险 1：平台侧反爬变化

越往“解析预览”前置，越依赖平台元数据提取稳定性。

但这个风险本来就存在，不是做了预览才有。区别只是你现在把它提前暴露出来了。

### 风险 2：`app.py` 继续膨胀

如果不先拆接口和 provider，预览功能大概率会继续把 `app.py` 堆胖。

所以这次做预览时，最好顺手把几个边界拆出来：

- input normalize
- source parse
- artifact preview
- provider dispatch

### 风险 3：需求范围膨胀

“预览”很容易越做越大，最后变成：

- 文件管理器
- 在线编辑器
- CMS

所以第一版要守住边界：

- 只读预览
- 复制 / 下载
- 不先做在线编辑

## 总体难度评估

如果按“MVP 够用”来做，我的判断是：

- `Phase 1 产物预览`：容易，应该尽快做
- `Phase 2 解析预览`：中等，是下一步关键
- `Phase 3 provider 抽象`：稍重，但长期必须
- `Phase 4 独立内容页`：中等偏产品化

如果只做一个有明显提升的版本，整体可控，属于：

- 研发难度：`中`
- 产品收益：`很高`
- 维护收益：`高`

## 推荐优先级

最推荐的顺序不是一下子全上，而是：

1. 先做 Web 内稿件预览
2. 再做 `POST /api/parse`
3. 再抽 provider
4. 最后做独立内容页

## 最后判断

这个需求不是“锦上添花”，而是一个很好的升级方向。

因为它会让幕库从：

- “把文件做出来”

走向：

- “把内容直接交到用户眼前”

这一步非常符合幕库的定位，而且比“单纯多接几个平台”更能拉开产品体验差异。
