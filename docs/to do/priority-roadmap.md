# 幕库需求优先级

这份文档不是单纯列想法，而是给后续开发做“逆向管理”的优先级底稿。

核心原则是：

- 先修会伤害用户信任的问题
- 再补最能提升核心体验的问题
- 最后再做平台扩张和架构升级

换句话说，先保证“说得对、用得通、跑得稳”，再追求“做得更大”。

## 优先级规则

### P0

必须优先处理。

满足任一条件就进 P0：

- 文档、UI、命令行为不一致，容易误导用户
- 新手最常见路径会踩坑
- 会直接破坏产品信任
- 不修就不适合继续宣传、扩平台、拉新用户

### P1

核心体验增强。

特点是：

- 不修不会立刻失真
- 但修完会明显提升幕库的产品完成度
- 会直接增强“Video-to-Markdown / Knowledge workspace”的定位

### P2

中长期建设项。

特点是：

- 很重要，但不是当前最卡脖子的地方
- 更偏底层架构、长期扩展和差异化能力

## 当前总判断

现在最适合的节奏不是先继续接更多平台，而是：

1. 先修真实能力和外部表达
2. 再补 Web 侧的核心消费体验
3. 再补长视频 ASR 稳定性
4. 最后再做更重的平台抽象和多平台扩张

## P0

## P0-1：README / docs / Web 文案对齐真实能力

状态：`已完成`

### 为什么是 P0

当前最明显的问题不是“功能没有”，而是“看起来像已经有了”。

典型误解包括：

- Web 看起来会默认产出 `知识库.md`
- Docker 看起来优先推荐 `*_COOKIES_FROM_BROWSER=chrome`
- `doctor` 看起来像“已经验证可用”，实际上更接近“已经配置”

这些问题会直接伤害用户信任。

### 范围

- `README.md`
- `docs/windows.md`
- Web 首页与设置页 copy
- 能力说明和产物说明

### 完成标准

- 新用户不会再误以为 Web 默认产出 `知识库.md`
- Docker 用户不会再被优先引导到高失败率的 browser cookies
- `doctor` 在文案层面不再被描述成 verified 检查

### 备注

这一项没做完之前，不建议继续强化“万能下载器 / 知识库一键完成”的外部表达。

## P0-2：Web 的知识库能力对齐真实行为

状态：`已完成`

### 为什么是 P0

现在 Web 的知识库配置已经暴露出来，但默认任务链路并不会真的生成 `知识库.md`。

这属于能力承诺和真实行为不一致。

### 决策方式

这一项先不急着直接加功能，先做决策：

- 方案 A：先把 Web 文案降级到真实状态
- 方案 B：直接补真正的 Web 知识库开关

### 推荐顺序

先 A，再 B。

### 完成标准

满足任一项即可先关闭这个 P0：

- Web 不再暗示默认会生成 `知识库.md`
- 或者 Web 真正支持显式开启知识库生成

### 后续延伸

如果选择补功能，知识库应作为显式任务开关，而不是“用户以为默认有”。

## P0-3：Docker 认证策略和 doctor 语义校正

状态：`已完成`

### 为什么是 P0

这属于最典型的新手踩坑路径：

- 文档推荐 browser cookies
- 但 Docker 环境里未必真的能读到浏览器配置
- `doctor` 又只检查有没有填变量

这会让用户误以为“已经配好了”，结果任务还是失败。

### 范围

- Docker 文档
- Windows 文档
- README 的认证建议
- `doctor` 输出文案

### 完成标准

- Docker 默认推荐 `cookies.txt`
- 本地 Python 运行时再优先推荐 `*_COOKIES_FROM_BROWSER`
- `doctor` 至少在文案和字段语义上区分 `configured` 与 `verified`

## P0 排序

建议顺序：

1. `P0-1` 文档 / UI 文案对齐
2. `P0-2` Web 知识库能力对齐
3. `P0-3` Docker 认证策略和 doctor 语义校正

如果要并行，`P0-1` 和 `P0-3` 可以一起做。

## P1

## P1-1：Web 产物预览

状态：`已完成`

### 为什么是 P1

这是幕库从“任务面板”走向“内容工作台”的第一步。

用户完成任务后，不应该只看到：

- 文件路径
- 状态
- 目录

而应该直接看到内容本身。

### 目标

Web 直接预览：

- `原始逐字稿.txt`
- `逐字稿.md`
- `解析稿.md`
- `知识库.md`
- `转写信息.json`

### 完成标准

- 用户不需要切去文件夹也能阅读单篇内容
- 能在 Web 中直接切换预览 `原始逐字稿 / 逐字稿 / 解析稿 / 知识库稿 / metadata`

### 判断

这是最值得优先做的 P1。

## P1-2：chunked ASR

状态：`已完成`

### 为什么是 P1

“抓不到字幕时还能直接出逐字稿”是幕库的核心价值之一。

现在最大的稳定性瓶颈不是解析不到标题，而是长视频整段 ASR 容易被远端截断。

### 目标

把 fallback 从“一整段音频一次性转写”升级为：

- 平台字幕直提
- 自动字幕
- 音频切片 ASR
- 最后才考虑 OCR 硬字幕

### 完成标准

- 长视频在无字幕场景下成功率明显提高
- 单段失败可以重试，不再因为一次截断全盘失败

### 判断

这是最关键的稳定性 P1。

## P1-3：源内容解析预览

状态：`下一步`

### 为什么是 P1

这一项决定幕库能不能从“提交任务”升级到“先判断内容值不值得处理”。

### 目标

新增 `POST /api/parse` 或等价能力，先返回：

- 标题
- 作者
- 平台
- 封面
- 时长
- 格式
- 字幕可用性
- 推荐处理链路

### 完成标准

- 用户粘贴链接后可以先看预览，再决定是否开始任务

### 判断

这是 Web 体验升级的关键 P1，但排在产物预览和 chunked ASR 后面。

## P1-4：逐字稿.md 资产化

状态：`待做`

### 为什么是 P1

当前 `逐字稿.md` 更像“清洗后的正文”，还不够像真正的 Markdown 资产。

而幕库的定位本来就是：

- 可检索
- 可连接
- 可被 AI 持续使用

### 目标

让 `逐字稿.md` 满足一个原则：

人能读、AI 能引用、出错能回溯。

### 建议内容

- 顶部元信息
- 轻量分段
- 可选时间锚点
- 保持和 `原始逐字稿.txt`、`解析稿.md`、`知识库.md` 的职责边界

### 判断

这是内容质量层面的重要 P1。

## P1-5：迁移到另一台电脑的使用文档

状态：`待做`

### 为什么是 P1

这不是最核心的产品功能，但对真实用户很重要，尤其是：

- 多台电脑切换
- Docker / 本地混用
- 下载目录、cookies、配置迁移

### 完成标准

- 有一份明确的迁移说明
- 用户知道什么可以直接复制，什么需要重新检查

## P1 排序

建议顺序：

1. `P1-3` 源内容解析预览
2. `P1-4` 逐字稿.md 资产化
3. `P1-5` 迁移文档

## P2

## P2-1：provider 抽象

状态：`待做`

包括：

- `yt-dlp` 默认 provider
- `Douyin` 专用 provider
- 后续更多平台统一走 provider 协议

这项长期非常重要，但不是当前最需要先解决的用户感知问题。

## P2-2：本地 ASR 后端

状态：`待做`

包括：

- `faster-whisper`
- `whisper.cpp`
- `mlx-whisper`

它们很有价值，尤其对离线和跨电脑稳定性有帮助，但当前优先级低于先把 chunked ASR 跑通。

## P2-3：独立内容页 / 单篇内容页

状态：`待做`

也就是把预览继续升级为真正的内容页面：

- 独立 URL
- 阅读页
- 更强的导出和分享能力

这是很好的方向，但不是当前最卡脖子的需求。

## P2-4：更多平台扩展

状态：`待做`

包括但不限于：

- 更多 `yt-dlp` 已支持平台
- 额外专用 provider
- 预览与解析的跨平台统一

在 P0 和 P1 没做稳之前，不建议把主要精力放在继续扩平台。

## 项目推进建议

如果我们后面按“逆向管理”来做，建议每次迭代都先回答两个问题：

1. 这次改动是在修信任，还是在补体验，还是在做长期架构？
2. 这次改动完成后，用户最核心的误解或阻塞有没有被消掉？

## 最终建议

如果从当前状态往后选两件事立刻做，我会选：

1. `P1-2`
2. 然后直接进 `P1-3`

也就是：

- 先把任务前的解析预览做出来
- 再把逐字稿资产质量继续做实
- 然后继续把幕库从“工作台”推进到“内容判断 + 处理”的完整闭环

如果只能选一件最影响“下一阶段产品完成度”的事，那就是：

- `P1-3` 源内容解析预览
