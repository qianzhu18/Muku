# 多平台整合评估

这份文档评估两个外部项目是否适合并入当前 `ytdlp-webui` / `video-downloade` 链路，用来扩大“分享链接 + 网址”输入覆盖范围：

- `jiji262/douyin-downloader`
- `ltaoo/wx_channels_download`

目标不是简单“缝合代码”，而是判断：

1. 是否适合纳入当前 Web + CLI + Docker + Skill 架构
2. 更适合做核心 provider、可选桥接，还是外部依赖
3. 对“视频 -> 逐字稿 -> 知识库”主链路的价值有多大

## 结论先行

### 抖音 `douyin-downloader`

适合整合，但更推荐做成独立 provider / adapter，而不是直接把整个仓库源码硬拷进来。

原因：

- 同样是 Python 工具，技术栈和当前项目更接近
- 已经支持短链、主页批量、收藏、合集、音乐、重试、SQLite 去重、Docker
- 对“更多平台分享链接和网址输入”这条路帮助很直接

建议定位：

- `DouyinProvider`
- 负责“链接解析 + 下载/导出音频或视频 + metadata”
- 产物继续走当前项目已有的逐字稿、清洗、解析稿、知识库链路

### 视频号 `wx_channels_download`

不适合直接并入当前核心后端，也不适合直接缝进 Docker 服务端。

更适合做“桌面端桥接器”或“外部下载前置步骤”。

原因：

- 它本质上依赖微信 PC 客户端、代理注入、证书安装、本地管理员权限
- 使用场景更偏“本机桌面操作增强”，不是通用后端下载器
- 当前许可证是 `MIT + Commons Clause`，商用和二次封装边界要更谨慎

建议定位：

- `WxChannelsBridge`
- 只负责“在用户本机桌面环境把视频号内容导出成可处理文件或下载命令”
- 然后把产物交给当前项目的 `audio` / `knowledge` 子命令继续整理

## 详细评估

## 1. 抖音 `jiji262/douyin-downloader`

### 适合整合的点

- Python 项目，和当前仓库的 CLI / Docker 形态更接近
- 已经覆盖短链、单视频、图集、合集、音乐、用户主页批量下载
- 自带 Docker、重试、并发、SQLite 去重、浏览器 fallback
- 还有可选转写配置，这说明它本身也接近“下载后继续处理”的工作流

### 不建议直接硬缝的点

- 它有自己完整的配置体系、目录结构、去重逻辑和任务模型
- 当前仓库已经有一套 `Job / sidecar / artifacts / knowledge` 契约
- 如果直接把两边 CLI 和配置文件混在一起，后面维护成本会明显上升

### 推荐接法

第一阶段：

- 只借它的“抖音链接解析 + 下载能力”
- 输出统一为当前项目的 `download_path + metadata`
- 后续全部复用当前仓库的逐字稿与知识库流程

第二阶段：

- 如果抖音 provider 稳定，再补抖音短链分享文案提取
- 再决定是否支持主页批量采集进知识库

### 最适合你的整合边界

- 不要先整仓 vendoring
- 优先做 provider interface
- 必要时把 `douyin-downloader` 当外部命令或子模块桥接

## 2. 视频号 `ltaoo/wx_channels_download`

### 适合参考的点

- 已经证明“视频号内容可下载”这件事有现实可行路径
- 维护活跃，最近还在持续适配样式变化和下载入口变化
- 对“桌面端分享后处理”很有启发价值

### 不适合直接整合进当前后端的点

- 它核心依赖微信 PC 客户端运行环境
- 首次启动要安装证书、启动本地代理、需要管理员权限
- 本质上是通过桌面端注入/拦截来拿资源，不是一个干净的服务端 URL 下载器
- Docker 容器、远程 Linux 服务器、纯 Web 后端都不是它的主战场

### 对当前项目的正确价值

它更像一个“上游采集桥”，而不是当前项目里的“平台 provider”。

也就是说更合理的链路是：

1. 用户本机通过 `wx_channels_download` 拿到视频或下载命令
2. 当前项目接收本地音频/视频文件
3. 再走 `audio --knowledge` 或后续的本地媒体入口

### 许可证层面的提醒

如果后面你想把它更深地嵌入产品、对外托管、商业化或 SaaS 化，务必要重新核对它的 `Commons Clause` 条件，而不是只看见 `MIT`。

## 3. 对当前架构的建议

最推荐的不是“按平台拼命塞代码”，而是先把平台层抽象出来。

推荐拆成三层：

### 输入层

- 原始分享文案
- 短链
- 直链 URL
- 本地音频/视频路径

### 平台 provider 层

- `YtDlpProvider`
- `DouyinProvider`
- `WxChannelsBridgeProvider`

### 统一后处理层

- 音频准备
- 字幕优先 / 音频转写
- 清洗稿
- 解析稿
- 知识库稿
- artifacts / sidecar

这样做的好处是：

- 新平台只改 provider，不用重写知识库链路
- CLI / Web / Docker / Skill 都能共享同一套输出契约
- 更适合后面继续接小红书、公众号文章、播客链接等输入

## 4. 推荐落地顺序

1. 先做输入规范化模块，把分享文案和短链提取独立出来
2. 抽象 provider interface
3. 第一优先级接 `DouyinProvider`
4. 视频号先做 bridge 方案，不要直接缝进核心下载器
5. 稳定后再考虑批量主页/收藏等高阶能力

## 5. 最终判断

- 抖音：值得整合，而且值得尽快做
- 视频号：值得接，但应该作为“桥接型能力”，不应该直接缝成核心后端 provider
- 如果你的目标是“抓取更多平台的分享链接和网址”，优先顺序应该是：
  1. Douyin
  2. 更多 `yt-dlp` 已覆盖平台
  3. 视频号桥接
