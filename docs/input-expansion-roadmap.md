# 输入链接与多端拓展路线

这份文档聚焦的是“输入层”能力，而不是下载或转写本身。

核心问题是：用户给你的不一定是标准 URL，可能是网页地址、短链、App 分享文本，甚至是安卓系统里的分享动作。项目如果想继续拓展，就需要先把输入层设计清楚。

## 目标

把当前“只接受直接 URL”的能力，扩展成“能稳定识别并规范化多种分享输入”。

第一阶段重点平台：

- Bilibili
- YouTube

## 建议的输入链路

建议把输入处理拆成一条独立流水线：

1. 接收原始输入
2. 提取候选链接
3. 展开短链或分享链接
4. 识别平台与内容类型
5. 生成规范化任务请求
6. 进入现有下载 / 转写流程

这样后面无论入口来自 Web、CLI、安卓分享还是浏览器插件，底层都能复用同一套逻辑。

## 建议支持的输入类型

### 第一层：标准网页链接

- `https://www.bilibili.com/video/BV...`
- `https://www.youtube.com/watch?v=...`
- `https://youtu.be/...`
- `https://www.youtube.com/shorts/...`

这是当前最容易先稳定支持的一层。

### 第二层：短链与分享链接

- `https://b23.tv/...`
- App 分享文本中的短链
- 带有额外追踪参数的移动端分享链接

这一步的重点不是下载，而是先把链接展开并清洗成标准 URL。

### 第三层：整段分享文案

常见情况是用户粘贴的不是 URL，而是一整段分享文字，例如：

- 视频标题 + 作者 + 短链
- “打开 App 查看更多内容”之类的分享文案
- 浏览器地址栏复制出的带很多参数的完整链接

这一步需要先做文本抽取，再做 URL 规范化。

### 第四层：系统级分享入口

这是安卓 APK 的关键能力：

- 从 Bilibili App 分享到本项目
- 从 YouTube App 分享到本项目
- 从浏览器“分享到...”进入本项目

这一层本质上不是新下载逻辑，而是给后端增加新的触发入口。

## 建议补出的模块边界

为了避免输入逻辑散落在 Web UI、CLI 和未来 APK 里，建议后续抽一层统一模块，例如：

- `input_normalizer`：负责从原始文本中提取 URL
- `share_link_resolver`：负责展开短链和移动端分享链接
- `platform_router`：负责识别 Bilibili、YouTube 等平台
- `job_payload_builder`：负责把规范化结果转成统一任务结构

这样现有的 `POST /api/start` 未来也更容易升级成更稳定的输入接口。

## 建议的阶段路线

### Phase 1：Web / CLI 输入规范化

先在现有仓库里补足这些能力：

- 标准 URL 识别
- `b23.tv` 与 `youtu.be` 规范化
- 去掉无关追踪参数
- 对无效输入给出更明确报错

这是最适合先落地的第一步，改动小，收益直接。

### Phase 2：分享文本识别

在 Web 和 CLI 中支持直接粘贴分享文案：

- 自动提取其中的主链接
- 兼容 App 复制出的整段文本
- 给用户显示“识别后的规范化链接”

这一步会明显提升移动端复制粘贴体验。

### Phase 3：统一 API 输入层

建议把现有 API 入口逐步升级成更清晰的结构，例如：

- 保留当前 `POST /api/start`
- 后续可新增一个偏输入层的接口，专门负责解析原始分享文本

这样 Web、CLI、安卓端都能依赖同一套后端协议。

### Phase 4：安卓 APK

安卓端第一版不建议做复杂的本地下载器，而是做一个轻量入口：

- 接收系统分享
- 展示识别后的链接
- 调用现有后端 API 发起任务
- 轮询任务状态并展示结果

这个版本的目标是“把分享入口打通”，不是立刻把全部桌面功能搬到手机里。

## 安卓 APK 的推荐实现顺序

### 方案 A：轻量壳层 App

第一版更推荐这个方案。

它的职责是：

- 接收 Android Share Intent
- 识别并展示链接
- 把任务提交给你的 Web 服务
- 显示任务状态和结果

优点：

- 开发成本最低
- 可以快速验证真实使用场景
- 不需要先把下载和转写全部搬到移动端本地运行

### 方案 B：移动端本地化能力

这是更后面的方向，不建议现在就做。

因为它会立刻引入：

- 手机端文件管理
- 本地下载权限
- 后台任务稳定性
- 模型和转写能力的资源限制

它更适合在后端链路稳定后再评估。

## 对开源文档的建议写法

面对 GitHub 社区，建议始终把输入层路线讲成三句话：

1. 当前已经支持直接 URL
2. 下一步会补分享链接和分享文案识别
3. 安卓端会优先做“分享到应用后发起解析任务”的轻量入口

这样方向很清楚，也不会给使用者造成“全部已经做好”的误解。
